# Task 6 — Repository слой: CRUD операции с базой данных

## Что такое Repository и зачем он нужен

Когда ты пишешь веб-приложение, рано или поздно встаёт вопрос: **где писать SQL-запросы?**

Можно писать их прямо в хендлерах (функциях, которые обрабатывают HTTP-запросы). Это работает, но быстро превращается в кашу — один файл отвечает и за HTTP, и за бизнес-логику, и за базу данных. Любое изменение в структуре БД ломает код в десятках мест.

**Repository паттерн** решает эту проблему. Это отдельный слой приложения, единственная задача которого — работать с базой данных. Хендлеры не знают про SQL. Они просто вызывают методы репозитория: "дай мне URL по коду", "сохрани эту запись".

Представь это как **кассира в банке**: ты не лезешь сам в хранилище, ты говоришь кассиру что нужно — он сам знает как это сделать.

## Структура таблицы в нашем проекте

Наш URL Shortener хранит данные в таблице `urls`:

```sql
CREATE TABLE IF NOT EXISTS urls (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    short_code  TEXT NOT NULL UNIQUE,
    original_url TEXT NOT NULL,
    created_at  DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

Что здесь происходит:

| Поле | Тип | Назначение |
|---|---|---|
| `id` | INTEGER | Уникальный номер записи, проставляется автоматически |
| `short_code` | TEXT | Короткий код ссылки, например `abc123`. UNIQUE — не может повторяться |
| `original_url` | TEXT | Оригинальный длинный URL |
| `created_at` | DATETIME | Дата создания, проставляется автоматически |

Из этой таблицы вытекают три операции, которые нужны нашему приложению:
- **Сохранить** пару `original_url → short_code`
- **Найти запись** по короткому коду (для редиректа)
- **Найти запись** по оригинальному URL (чтобы не создавать дубликаты)

## Интерфейс и инкапсуляция

Посмотри на наш файл `internal/repository/url-repository.go`. В нём два ключевых типа:

```go
// Интерфейс — публичный контракт. Описывает ЧТО умеет репозиторий.
type URLRepository interface {
    Save(url *model.URL) error
    GetByShortCode(shortCode string) (*model.URL, error)
    GetByOriginalURL(originalURL string) (*model.URL, error)
}

// Структура — приватная реализация. Описывает КАК именно это делается.
type sqliteURLRepository struct {
    db *sql.DB
}
```

### Почему структура написана с маленькой буквы?

В Go регистр первой буквы определяет видимость:
- `URLRepository` — **заглавная буква** — публичный тип, виден снаружи пакета
- `sqliteURLRepository` — **строчная буква** — приватный тип, виден только внутри пакета `repository`

Это означает, что никто снаружи не может напрямую создать `sqliteURLRepository`. Попытка написать такой код приведёт к ошибке компиляции:

```go
// main.go — это НЕ скомпилируется:
repo := repository.sqliteURLRepository{}  // ошибка: тип недоступен
```

### Как тогда получить репозиторий снаружи?

Через конструктор `NewURLRepository`:

```go
func NewURLRepository(db *sql.DB) URLRepository {
    return &sqliteURLRepository{db: db}
}
```

Обрати внимание: функция возвращает тип `URLRepository` (интерфейс), а не `*sqliteURLRepository` (конкретную структуру). Это принципиально.

**Что это даёт?** Вызывающий код — хендлеры, `main.go` — знает только о трёх методах: `Save`, `GetByShortCode`, `GetByOriginalURL`. Он понятия не имеет, что внутри используется SQLite. Завтра ты можешь написать `postgresURLRepository`, которая реализует тот же интерфейс, — и хендлеры не изменятся вообще.

## Как структура реализует интерфейс в Go

В Go нет ключевого слова `implements` как в Java или C#. Структура **автоматически** считается реализующей интерфейс, если у неё есть все методы с правильными сигнатурами.

`sqliteURLRepository` реализует `URLRepository`, потому что у неё есть:
- метод `Save(url *model.URL) error` ✓
- метод `GetByShortCode(shortCode string) (*model.URL, error)` ✓
- метод `GetByOriginalURL(originalURL string) (*model.URL, error)` ✓

Методы "прикреплены" к структуре через **receiver** — `(sURLRep *sqliteURLRepository)`:

```go
func (sURLRep *sqliteURLRepository) Save(url *model.URL) error {
    // sURLRep — это экземпляр структуры, как self/this в других языках
}
```

## Методы и SQL-запросы

### Save — сохранить запись

```go
func (sURLRep *sqliteURLRepository) Save(url *model.URL) error {
    _, err := sURLRep.db.Exec(
        "INSERT INTO urls (short_code, original_url) VALUES (?, ?)",
        url.ShortCode,
        url.OriginalURL,
    )
    if err != nil {
        return fmt.Errorf("Ошибка при записи данных в БД: %w", err)
    }
    return nil
}
```

- `db.Exec()` — выполняет запрос, который **не возвращает строки** (INSERT, UPDATE, DELETE)
- `?` — это плейсхолдеры, значения подставляются автоматически. Это защита от SQL-инъекций: никогда не собирай SQL-запрос конкатенацией строк
- `_` — мы игнорируем первый возврат (`sql.Result`), он нам здесь не нужен
- Если в базе уже есть запись с таким `short_code` (нарушение UNIQUE), `err` будет не `nil`

### GetByShortCode — найти по короткому коду

```go
func (sURLRep *sqliteURLRepository) GetByShortCode(shortCode string) (*model.URL, error) {
    url := &model.URL{}
    err := sURLRep.db.QueryRow(
        "SELECT id, short_code, original_url, created_at FROM urls WHERE short_code = ?",
        shortCode,
    ).Scan(&url.ID, &url.ShortCode, &url.OriginalURL, &url.CreatedAt)
    if err != nil {
        return nil, fmt.Errorf("Ошибка при запросе: %w", err)
    }
    return url, nil
}
```

- `db.QueryRow()` — выполняет запрос, который возвращает **одну строку**
- `.Scan()` — считывает значения из этой строки и записывает их в переменные

### GetByOriginalURL — найти по оригинальному URL

Этот метод нужен для **проверки дубликатов**. Перед тем как сгенерировать новый короткий код, сервис может спросить репозиторий: "А вдруг этот URL уже есть в базе?" Если есть — вернуть существующий код, а не плодить новые. Реализация аналогична `GetByShortCode`, только условие WHERE другое.

## Почему в Scan() передают `&` (амперсанд)

Это один из самых частых вопросов у начинающих Go-разработчиков.

`.Scan()` — это функция, которая **записывает** данные из строки базы данных в твои переменные. Чтобы функция могла изменить переменную, ей нужен **адрес этой переменной в памяти**, то есть указатель.

### Простая аналогия

Представь, что ты хочешь попросить друга написать что-то в твоём блокноте:
- Если ты скажешь другу номер страницы (адрес) — он найдёт блокнот и запишет. ✓
- Если ты просто скажешь "запиши куда-нибудь" — он запишет в своей копии, а твой блокнот останется пустым. ✗

`&` — это оператор взятия адреса. `&url.ID` означает "адрес поля ID в памяти".

### Что происходит без `&`

```go
// Передаём значение (копию) — Scan запишет в копию, оригинал не изменится
.Scan(url.ID, url.ShortCode, ...)  // url останется пустым

// Передаём адрес — Scan запишет по адресу, оригинал изменится
.Scan(&url.ID, &url.ShortCode, ...) // url заполнится данными из БД ✓
```

### Порядок важен

```go
.Scan(&url.ID, &url.ShortCode, &url.OriginalURL, &url.CreatedAt)
//     ^         ^               ^                 ^
//     id        short_code      original_url      created_at
//     (1-я колонка в SELECT)    (3-я колонка)     (4-я колонка)
```

Порядок аргументов в `Scan()` должен **точно совпадать** с порядком колонок в `SELECT`. Первый аргумент получает значение первой колонки, второй — второй, и так далее.

## Важный нюанс: обработка "запись не найдена"

В текущей реализации есть место для улучшения. Когда запись не найдена, `QueryRow().Scan()` возвращает специальную ошибку `sql.ErrNoRows`. Сейчас она обрабатывается так же, как любая другая ошибка:

```go
if err != nil {
    return nil, fmt.Errorf("Ошибка при запросе: %w", err)
}
```

Проблема в том, что хендлер не может отличить:
- "Ссылка не найдена" → нужно вернуть клиенту **404 Not Found**
- "База данных упала" → нужно вернуть клиенту **500 Internal Server Error**

В боевом коде `sql.ErrNoRows` обрабатывают отдельно:

```go
if err == sql.ErrNoRows {
    return nil, nil  // или специальная ошибка типа ErrNotFound
}
if err != nil {
    return nil, fmt.Errorf("Ошибка при запросе: %w", err)
}
```

## Итоговая картина: как слои взаимодействуют

```
HTTP Request
     ↓
  Handler          — знает про HTTP, не знает про SQL
     ↓
URLRepository      — интерфейс (контракт)
     ↓
sqliteURLRepository — знает про SQL и SQLite
     ↓
  База данных
```

Каждый слой знает только о соседнем. Хендлер зависит от интерфейса, а не от конкретной реализации. Это называется **разделение ответственности** (Separation of Concerns) — один из фундаментальных принципов проектирования программ.
