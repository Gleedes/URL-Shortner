День 1: Архитектурная Ревизия
Прочитать текущие файлы: main.go, config.go, url.go, repository, service, migration.
Выписать ответственность каждого слоя.
Зафиксировать целевую схему: main собирает зависимости, handler принимает HTTP, service решает бизнес-задачу, repository работает с БД.
Найти текущие архитектурные проблемы: service знает про sql.ErrNoRows, short code пустой, ошибок сохранения не видно, HTTP API почти нет.


1. В main.go мы запускаем сервер, который слушает порт 8080.
2. config.go - место обработки констант, которые храняться в .env, все храниться в одном месте, легко получить к нужному значению.
3. Тут храняться модели данных, пока что только структура URL, она хранит поля, которые необходимы для реализации основной идеи сервиса
4. Слой repository - работает с данными. В моем случае это SQLite. В нем мы создаем методы, которые напрямую взаимодействуют с базой данных через CRUD-операции. Всего три метода: save(), запись пары URL-shortCode, getByShortCode() - вывести запись по короткому коду, getByOriginalURL() - вывести запись по оригинальному URL. Также реализован конструктор, который возвращает тип интерфейса, чтобы скрыть от внешних слоев реализовацию структуры sqliteURLRepository
5. Слой Service - здесь реализована бизнес логика приложения, все правила, вализации, проверки - все тут. Должны быть реализованы функции сокращения ссылки, введение счета, проверка на коллизии, обработка ошибок.
6. Слой Handler - он работает по принципу. "запрос-ответ". Он знает, какое use case правило нужно вызвать, чтобы обработать запрос, но сам не обрабатывает его.
